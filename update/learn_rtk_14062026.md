What RTK teaches us
1. Terminal output is a first-class context problem

Your current template controls what files the agent reads. RTK controls what command output the agent sees.

That matters because an agent session usually wastes tokens in two places:

Waste source	Current template handles?	RTK handles?
Reading too many files	Yes	No
Reading huge test output	Not really	Yes
Reading noisy git status / git diff	Partly	Yes
Reading noisy find, grep, ls, tree	Partly	Yes
Reading large logs	Not really	Yes
Re-running commands and polluting context	Not really	Yes

RTK’s architecture is basically: parse command → route → execute → filter → print → track savings. Its architecture docs emphasize exit-code preservation, fail-safe fallback to original output, and transparent raw-output access with verbosity flags.

That is exactly the kind of deterministic tool layer an AI template should have. A good AI product is not only the model; it includes context building, retrieval, tool use, validation, analytics, and monitoring.

What we should apply into your template
A. Add docs/agent/COMMAND_OUTPUT_POLICY.md

This should define when agents use compressed output vs raw output.

Recommended content:

# Command Output Policy

Use compressed command output by default for noisy commands.

Prefer RTK for:
- git status
- git diff
- git log
- grep / find / ls / tree
- pytest / vitest / cargo test / go test
- lint / typecheck
- docker / kubectl / logs

Use raw output when:
- debugging parser/filter behavior
- a test failure is unclear
- exact stdout/stderr matters
- reviewing security-sensitive output
- verifying generated artifacts
- the compressed result looks suspicious

Fallback:
- If RTK is unavailable, run the original command.
- If compressed output is insufficient, rerun with raw output.

RTK docs explicitly describe graceful degradation: hooks should not block execution if RTK is missing or filtering fails, and raw command output can still run unchanged.

B. Update AGENTS.md

Your current AGENTS.md already says agents should avoid scanning the whole repo, prefer rg, targeted reads, and module cards, and summarize changed files, commands, and risks. Add this block:

## Command Output Rules

For noisy terminal commands, prefer compressed output.

Use RTK when available:
- `rtk git status`
- `rtk git diff`
- `rtk grep`
- `rtk find`
- `rtk pytest`
- `rtk cargo test`
- `rtk npm test`
- `rtk tsc`
- `rtk eslint`

If compressed output is incomplete or suspicious:
1. rerun the specific command in raw mode;
2. inspect only the relevant failing section;
3. mention the rerun in the final command summary.

Do not hide failures. Test failures, stack traces, exit codes, and actionable errors must remain visible.

RTK is optional but recommended for token-efficient command execution.

### D. Add Makefile wrappers

Your template already has `make test-unit`, `make lint`, `make typecheck`, and agent-audit commands. :contentReference[oaicite:8]{index=8} Add RTK-aware wrappers without making RTK mandatory:

```makefile
RTK := $(shell command -v rtk 2>/dev/null)

define maybe_rtk
$(if $(RTK),rtk $(1),$(1))
endef

.PHONY: rtk-gain git-status git-diff test-unit-compact lint-compact typecheck-compact

rtk-gain:
	@if command -v rtk >/dev/null 2>&1; then rtk gain; else echo "rtk not installed"; fi

git-status:
	$(call maybe_rtk,git status)

git-diff:
	$(call maybe_rtk,git diff)

test-unit-compact:
	$(call maybe_rtk,pytest)

lint-compact:
	$(call maybe_rtk,ruff check .)

typecheck-compact:
	$(call maybe_rtk,mypy .)

The key design choice: RTK improves agent UX, but should not become a hard build dependency.
